rok

An AI-native code execution engine. Write one JSON payload — rok executes everything and returns one structured result.

24 Step Types Rust-powered ~90% Token Reduction Parallel Execution v0.10.0

Overview

rok collapses multi-step coding tasks into a single JSON document. Instead of an AI agent making 20+ tool calls to rename a function across a codebase, it writes one JSON payload — rok handles all file operations, bash commands, conditionals, and loops, then returns a single structured result.

The Core Principle Your job is planning (expressed as JSON). rok's job is execution. 
{
  "name": "my-task",
  "steps": [
    { "type": "grep", "pattern": "TODO", "path": "./src" },
    { "type": "bash", "cmd": "echo Found todos" },
    { "type": "write", "path": "todos.txt", "content": "..." }
  ]
}
Without rokWith rok
20+ API calls to rename a function1 JSON payload with a refactor step
Read files one-by-one to find TODOs1 grep step → each loop
Sequential bash commands across turns1 parallel step block
No rollback if changes break thingssnapshot before, restore on failure

Quick Start

Run inline JSON

rok -j '{"steps":[{"type":"bash","cmd":"echo hello"}]}'

Run from file

rok -f task.json

Pipe from stdin

echo '{"steps":[{"type":"bash","cmd":"ls"}]}' | rok

Pretty output

rok -f task.json -o pretty

Dry run (preview steps)

rok -f task.json --dry-run

Verbose / Quiet

rok -f task.json --verbose   # detailed output
rok -f task.json -q          # suppress output (errors only)

Payload Structure

{
  "name": "task-name",           // optional: task identifier
  "description": "What it does", // optional
  "version": "1.0.0",            // optional
  "author": "your-name",         // optional
  "options": {
    "cwd": ".",                  // working directory (default: ".")
    "stop_on_error": true,       // halt on first failure (default: true)
    "timeout_ms": 30000,         // global step timeout in ms
    "cache": false,              // enable result caching
    "cache_dir": ".rok/cache",   // custom cache directory
    "incremental": false,        // skip steps with unchanged inputs
    "env": {                     // extra environment variables
      "NODE_ENV": "production",
      "DEBUG": "false"
    }
  },
  "props": {
    "feature_name": "users",     // reusable variables across steps
    "component": "Button"
  },
  "steps": [
    // ... step objects
  ]
}

Common Step Fields

Every step supports these optional fields:

FieldTypeDescription
idstringNamed identifier for this step (used in depends_on and refs)

timeout_msnumberOverride global timeout for this step

File Operations

bash — Run Shell Commands

{
  "type": "bash",
  "id": "build",
  "cmd": "npm run build",
  "timeout_ms": 60000,
  "retry": {
    "count": 3,          // number of retries (default: 3)
    "delay_ms": 2000,    // delay between retries (default: 1000)
    "backoff": true      // exponential backoff
  }
}
FieldRequiredDescription
cmdyesShell command to execute
timeout_msnoTimeout in milliseconds
retrynoRetry configuration object

Output: { "stdout": "...", "stderr": "...", "exit_code": 0 }

read — Read Files

{
  "type": "read",
  "path": "src/**/*.ts",     // file path or glob pattern
  "max_bytes": 1048576,       // max bytes per file (optional)
  "encoding": "utf-8",        // file encoding (optional)
  "filter_imports": "React",  // only files importing this symbol
  "filter_exports": "Button", // only files exporting this symbol
  "since": "2024-01-01"       // only files modified after this date
}
FieldRequiredDescription
pathyesFile path or glob pattern
max_bytesnoMaximum bytes to read per file
filter_importsnoOnly return files that import this symbol
filter_exportsnoOnly return files that export this symbol
sincenoOnly files modified after this ISO date

Output: { "files": [{ "path": "...", "content": "..." }], "files_filtered": N }

write — Write File

{
  "type": "write",
  "path": "src/components/Button.tsx",
  "content": "export const Button = () => <button>Click</button>;",
  "create_dirs": true   // create parent directories if needed (default: true)
}

patch — Surgical Edits

Apply multiple find/replace edits to a single file atomically.

{
  "type": "patch",
  "path": "src/main.rs",
  "edits": [
    { "find": "fn old_name()",  "replace": "fn new_name()" },
    { "find": "let x = 1;",    "replace": "let x = 2;" },
    { "find": "use old::Mod;",  "replace": "use new::Mod;" }
  ]
}

Output: { "edits_applied": N, "diff": "..." }

mv / cp / rm / mkdir

{ "type": "mv",    "from": "old.txt",    "to": "new.txt" }
{ "type": "cp",    "from": "src/",       "to": "backup/", "recursive": true }
{ "type": "rm",    "path": "dist/",      "recursive": true }
{ "type": "mkdir", "path": "src/components/auth" }

Search & Replace

grep — Search Pattern

{
  "type": "grep",
  "pattern": "TODO|FIXME",     // search pattern (string or regex)
  "path": "./src",              // directory or file to search
  "ext": ["ts", "tsx", "js"],  // file extensions to include
  "regex": true,                // treat pattern as regex (default: false)
  "context_lines": 2            // lines of context around matches
}

Output: { "matches": [{ "path": "...", "line": N, "text": "..." }] }

Tip Use grep results with each to loop over matched files: { "ref": 0, "pick": "matches[*].path" }

replace — Find/Replace Across Files

{
  "type": "replace",
  "pattern": "oldFunction",      // what to find
  "replacement": "newFunction",  // what to replace with
  "path": "./src",               // root directory
  "ext": ["ts", "js"],           // file extensions (optional)
  "glob": "**/*.{ts,tsx}",       // glob pattern to filter files (optional)
  "regex": false,                // treat pattern as regex (default: false)
  "case_sensitive": true,        // case-sensitive (default: true)
  "whole_word": true             // only match whole words (default: false)
}

Output: { "files_scanned": N, "files_modified": M, "total_replacements": K }

refactor — Symbol Renaming v5 New

Rename a function, variable, class, or any symbol across your entire codebase with whole-word matching and optional dry-run preview.

{
  "type": "refactor",
  "symbol": "getUserById",         // current symbol name
  "rename_to": "fetchUserById",    // new symbol name
  "path": "./src",                 // root directory to search
  "ext": ["ts", "tsx", "js"],      // file extensions (optional)
  "whole_word": true,              // only match whole words (default: true)
  "dry_run": false,                // preview only, don't write files
  "preview": false                 // alias for dry_run
}
FieldRequiredDescription
symbolyesThe symbol name to rename
rename_toyesThe new symbol name
pathyesDirectory to search in
extnoFile extensions to include
whole_wordnoOnly match whole words (default: true)
dry_runnoPreview changes without writing (default: false)

Output:

{
  "symbol": "getUserById",
  "rename_to": "fetchUserById",
  "files_scanned": 45,
  "files_modified": 8,
  "total_replacements": 12,
  "dry_run": false,
  "changes": [
    { "path": "src/api.ts", "line": 23, "old_text": "getUserById(id)", "new_text": "fetchUserById(id)" }
  ]
}
Safe Rename Pattern Always snapshot before a large refactor, then dry_run to preview, then apply: 
{
  "steps": [
    { "type": "snapshot", "path": "src", "snapshot_id": "pre-rename" },
    { "type": "refactor", "symbol": "oldName", "rename_to": "newName", "path": "./src", "dry_run": true },
    { "type": "if",
      "condition": { "type": "stepOk", "ref": 1 },
      "then": [{ "type": "refactor", "symbol": "oldName", "rename_to": "newName", "path": "./src" }]
    }
  ]
}

Code Intelligence

scan — Project Map

{
  "type": "scan",
  "path": "./src",
  "depth": 3,
  "include": ["ts", "tsx"],
  "output": "full"   // "summary" | "full" | "imports" | "exports"
}

Output: { "tree": {...}, "exports": {...}, "imports_graph": {...}, "file_count": N }

summarize — File Structure

{
  "type": "summarize",
  "path": "./src/App.tsx",
  "focus": "exports"   // "exports" | "imports" | "" (all)
}

Output:

{
  "imports": ["react", "react-router"],
  "exports": ["App", "default"],
  "functions": ["handleClick", "renderHeader"],
  "types_used": ["FC", "useState"],
  "line_count": 142,
  "last_modified": "2026-03-15T10:22:00Z"
}

extract — Pull Keys from Config

Extract specific keys from JSON, TOML, or YAML config files without reading the whole file.

{
  "type": "extract",
  "path": "package.json",
  "pick": ["name", "version", "dependencies.react"]
}

Output: { "data": { "name": "my-app", "version": "1.0.0", ... } }

deps — Dependency Graph v5 New

Build a map of import/export relationships across your codebase. Supports JS/TS, Python, Rust, and Go.

{
  "type": "deps",
  "path": "./src",
  "depth": 4,
  "include": ["ts", "tsx"],  // file extensions to include
  "focus": "src/auth/index.ts"  // focus on a specific file (optional)
}
FieldRequiredDescription
pathyesRoot directory to analyze
depthnoDirectory traversal depth (default: 3)
includenoFile extensions to include
focusnoLimit graph to a specific file and its neighbors

Output:

{
  "path": "./src",
  "file_count": 42,
  "graph": {
    "src/app.ts": ["react", "src/auth", "src/api"],
    "src/auth/index.ts": ["react", "src/utils"]
  },
  "dependents": {
    "src/auth/index.ts": ["src/app.ts", "src/dashboard.ts"]
  },
  "cycles": [
    ["src/a.ts", "src/b.ts"]
  ]
}
Use deps before refactoring Run deps first to understand what files depend on what you're changing. Use focus to zoom in on a specific module.

import — Manage Imports

Add, remove, and organize imports in JS/TS, Python, Rust, or Go files.

{
  "type": "import",
  "path": "src/main.ts",
  "add": [
    "import { useState } from 'react';",
    "import { useCallback } from 'react';"
  ],
  "remove": [
    "import { oldHook } from 'react';"
  ],
  "organize": true   // sort and group imports
}

Output: { "added": [...], "removed": [...], "organized": true }

snapshot / restore — Rollback Safety

Create a point-in-time snapshot of files before risky changes. Restore if anything goes wrong.

// Create snapshot
{ "type": "snapshot", "path": "src/", "snapshot_id": "before-refactor" }

// Restore snapshot
{ "type": "restore", "snapshot_id": "before-refactor" }

checkpoint — Save/Resume State v5 New

Save execution state to disk and resume from a specific point. Useful for long-running multi-phase tasks.

// Save a checkpoint
{
  "type": "checkpoint",
  "checkpoint_id": "phase-1-complete",
  "restore": false   // false = save (default), true = restore
}

// Restore a checkpoint
{
  "type": "checkpoint",
  "checkpoint_id": "phase-1-complete",
  "restore": true
}
FieldRequiredDescription
checkpoint_idyesUnique identifier for the checkpoint
restorenoIf true, restore; if false, save (default: false)

Checkpoints are stored in .rok/checkpoints/{id}.json. List/delete via CLI: rok checkpoints --list

Control Flow

if — Conditional Execution

{
  "type": "if",
  "condition": { "type": "grepHasResults", "ref": 0 },
  "then": [
    { "type": "bash", "cmd": "echo 'Found issues'" },
    { "type": "bash", "cmd": "npm run fix" }
  ],
  "else": [
    { "type": "bash", "cmd": "echo 'All clean'" }
  ]
}

Condition Types

TypeFieldsDescription
existspathFile or directory exists
containspath, pattern, regex?File content matches pattern
grepHasResultsrefReferenced grep step has matches
stepOkrefReferenced step succeeded
stepFailedrefReferenced step failed
fileChangedpath, sinceFile modified after date
notconditionNegate a condition
andconditionsAll conditions must be true
orconditionsAny condition must be true

each — Loop Over Items

{
  "type": "each",
  "over": ["file1.ts", "file2.ts", "file3.ts"],  // static list
  "as": "file",
  "parallel": true,    // run iterations in parallel (default: true)
  "step": {
    "type": "summarize",
    "path": "{{file}}"
  }
}

// Or loop over ref output
{
  "type": "each",
  "over": { "ref": 0, "pick": "matches[*].path" },
  "as": "file",
  "step": { "type": "read", "path": "{{file}}" }
}

parallel — Concurrent Execution

{
  "type": "parallel",
  "steps": [
    { "type": "bash", "cmd": "npm run build:client" },
    { "type": "bash", "cmd": "npm run build:server" },
    { "type": "bash", "cmd": "npm run build:worker" }
  ]
}

Ref System

Steps can reference outputs from previous steps using numeric or named references.

Reference by Index

{
  "steps": [
    { "type": "grep", "pattern": "TODO", "path": "./src" },
    {
      "type": "bash",
      "cmd": "echo Found {{ref:0.matches.length}} TODOs"
    }
  ]
}

Reference by ID

{
  "steps": [
    { "type": "grep", "id": "find-todos", "pattern": "TODO", "path": "./src" },
    {
      "type": "each",
      "over": { "ref": "find-todos", "pick": "matches[*].path" },
      "as": "file",
      "step": { "type": "read", "path": "{{file}}" }
    }
  ]
}

Step Dependencies

Use id and depends_on to control execution order. rok builds a dependency graph and respects it.

{
  "steps": [
    { "type": "bash", "id": "install",  "cmd": "npm install" },
    { "type": "bash", "id": "build",    "cmd": "npm run build",    "depends_on": ["install"] },
    { "type": "bash", "id": "test",     "cmd": "npm test",         "depends_on": ["install"] },
    { "type": "bash", "id": "deploy",   "cmd": "npm run deploy",   "depends_on": ["build", "test"] }
  ]
}

In this example, build and test run in parallel after install, and deploy waits for both.

Conditions

// File exists
{ "type": "exists", "path": "./Cargo.toml" }

// File contains pattern
{ "type": "contains", "path": "src/main.ts", "pattern": "TODO", "regex": false }

// Previous grep step found something
{ "type": "grepHasResults", "ref": 0 }

// Step succeeded / failed
{ "type": "stepOk",     "ref": "build" }
{ "type": "stepFailed", "ref": "build" }

// Compound conditions
{ "type": "and", "conditions": [
    { "type": "exists", "path": "./Cargo.toml" },
    { "type": "stepOk", "ref": "fmt" }
]}

{ "type": "or", "conditions": [
    { "type": "exists", "path": "./bun.lockb" },
    { "type": "exists", "path": "./package-lock.json" }
]}

{ "type": "not", "condition": { "type": "exists", "path": "dist/" } }

Incremental Mode v5 New

Enable incremental mode to skip steps whose file inputs haven't changed since the last run. rok tracks file modification times in .rok/incremental_state.json.

{
  "name": "lint-changed-files",
  "options": {
    "incremental": true
  },
  "steps": [
    { "type": "grep", "pattern": "console.log", "path": "./src" },
    { "type": "lint", "path": "./src" }
  ]
}

On the first run, all steps execute. On subsequent runs, steps that operate on unchanged files are automatically skipped with status "skipped".

Incremental + Cache Combine incremental mode with caching for maximum efficiency: "incremental": true, "cache": true

Steps that support incremental skipping: read, grep, replace, scan, refactor, deps

Caching

Enable caching to skip re-running identical steps on repeated executions. Cache keys are hashed from the step JSON.

{
  "options": {
    "cache": true,
    "cache_dir": ".rok/cache"   // default
  },
  "steps": [...]
}
